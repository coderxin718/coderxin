云农智能体

面向种植户和农业科研人员的垂直领域智能问答系统，解决通用大模型在农业场景下专业知识不足、回答缺少来源及图像结果难以解读等问题。

项目概述

云农智能体是一个面向农业领域的智能问答系统，基于 Spring Boot 3.5 + LangChain4j + pgvector 构建，采用 ReAct Agent 多工具编排架构，通过阿里云 DashScope（Qwen-Max）大模型驱动。系统与植物表型分析平台联动，形成"图像感知 → 参数提取 → 知识检索 → 诊断报告"的完整闭环。

举个例子：种植户拍了一张水稻叶片发黄的照片，系统会（1）调用表型平台分析 NDVI/株高等参数 →（2）与正常标准比对判断是否异常 →（3）在病害图谱中检索匹配的症状和防治方案 →（4）生成一份带来源引用的诊断报告。整个过程由 Agent 自主决策执行，用户只需要发一张图片。

技术架构

接入层 — Controller，5 个端点：同步对话、SSE 流式、图像联动诊断、知识入库、文档上传
编排层 — ReAct Agent + Skill 双模式，Agent 自主决策，Skill 固定执行链
工具层 — 5 个农业 Tool（知识检索、病害图谱、种植规范、表型标准、图像分析）+ MCP 动态工具
数据层 — pgvector 向量+BM25 关键词双路混合检索 + Redis 会话记忆 + 表型平台 HTTP API

核心技术实现

1. ReAct Agent 多工具编排

基于 LangChain4j 的 AiServices 机制，通过接口 + 注解声明 Agent。注册 7 个工具（5 个农业专用 + 时间工具 + 检索工具），Agent 自主判断"当前需要哪个工具、传什么参数"。新增工具只需加一个 @Tool 注解的类，不需要改任何流程代码。

2. Tool + Skill 双模式

Tool 模式：LLM 自主判断要不要调、什么时候调、调几次，适用于低频灵活问答（如"水稻叶子发黄怎么办"）。Skill 模式：固定执行链，不依赖 LLM 逐步推理，适用于高频标准操作（如"分析这张水稻图片"直接走 T5→T4→T2 链路）。灵活性归 Tool，确定性归 Skill，两者互补。

3. 混合检索（向量 + BM25 双路召回）

纯向量搜索在专有名词上存在短板——"三环唑"和"三唑酮"语义相近但化学结构不同。系统采用双路召回 + Reranker 融合排序：BM25 关键词搜索精确命中专有名词，向量搜索理解"叶子发黄→叶片黄化"的语义映射，Reranker（Cross-Attention 模型）融合双路结果取各自长处。

4. 差异化文档切分 + 运行时入库

不是所有文档一刀切。学术论文按语义段落（~800字/块，100字重叠），种植规程按农事环节章节，病害图谱一个病害一个完整块，表型标准一个指标一个块。四种策略各自实现统一接口，配合上传接口实现运行时入库，知识库从静态快照变成持续生长的活系统。

5. 三层幻觉控制 + Evaluator Agent

入库层：每个 chunk 绑定 source_doc、page_num、chunk_id 元数据。生成层：系统提示词约束要求标注来源。后处理层：ResponseValidator 正则校验格式（毫秒级，全覆盖）+ Evaluator Agent 逐条对照检索原文审查事实真伪和来源真实性（秒级，仅高风险回答触发），发现问题发回修正，最多三轮。

6. SSE 多事件结构化流式

不是纯文字 token 一个个蹦出来。输出分为 thinking / tool_call / tool_result / message / error 五种结构化事件，前端根据事件类型做不同渲染，用户能感知"系统在思考"还是"正在查资料"还是"在回答"。

7. MCP 协议动态工具扩展

工具不再是写死在代码里的。已接入智谱 BigModel 网页搜索 MCP Server，已设计天气数据和土壤数据 MCP Server。外部数据服务只需实现 MCP Server 协议，Agent 就能自动发现和调用，彻底解耦服务的提供方和消费方。
